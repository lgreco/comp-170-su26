## `find` vs `index`: Two Ways to Search a String

We opened with Bella's question from the homework: when should we use `.find()` versus `.index()`? We explored this live with the string `"Chicago"`. Both methods locate a character and return its position — `"Chicago".index("i")` and `"Chicago".find("i")` both give `2`. The difference shows up when the character isn't there. Asking for `"z"` with `.index()` causes a traceback — the program crashes. Asking for `"z"` with `.find()` returns $-1$, a position that doesn't exist within the string, which is its way of saying "not found."

## Why Two Methods with Such Different Behavior?

The dramatic crash from `.index()` is a design choice: it signals that the caller made a promise — "this thing is definitely in here" — and broke it. When missing data represents a genuine, unrecoverable error, crashing loudly makes sense. `.find()` takes a gentler approach: it returns $-1$ and lets the program decide what to do next. For everyday string searching, where we can't always guarantee the target exists, `.find()` is the safer tool.

There's also a scope difference: `.find()` works only on strings. `.index()` works on strings and lists. We demonstrated this with a list of travel destinations and watched `.index("Rome")` crash when Rome wasn't on the list.

## Writing Our Own `indexOf`

We wrote a `contains` method that scans a list and returns `True` or `False`, then a custom `indexOf` method that returns $-1$ when the target is absent — the same convention used by most other languages. With our own method, we no longer need an `if` guard before calling `.index()`: the method handles the missing-value case itself.

## Python's Negative Indexing

One wrinkle: in Python, $-1$ is not just a sentinel for "not found" — it's a valid index meaning the last item in a list or string. `"Arlo"[-1]` returns `"o"`. Most other languages would reject a negative index outright. This is why using $-1$ as a "not found" value can be confusing in Python, and why writing your own `indexOf` with clearly documented behavior is a reasonable approach.

## Designing for Errors

We closed with a broader design principle: good programs anticipate user mistakes rather than exploding at the first sign of trouble. Accepting a plausible but wrong date of birth, flagging it for follow-up, is more useful than crashing. The right level of protection — and when to accept, warn, or reject — is a design conversation that happens before anyone writes code.
