## Multi-Branch If Statements for the ATM Withdrawal
We returned to last week's ATM withdrawal problem to explore why a multi-branch if/elif/else statement works better than two separate if statements for checking a withdrawal. With two separate if statements, the code would still go on to try the withdrawal even after printing an error, because nothing stops it from falling through to the next step. Bella asked about this difference directly, and together we worked through a nested if/else version to show how the else branches catch the same cases that elif would. The withdrawal only goes through if the amount is a multiple of 20 and does not exceed the account balance.

## Raising Errors vs. Giving the User Another Chance
We compared two approaches to handling bad input: raising a value error right away, which stops the program, versus printing a message and letting the person try again. Leo argued for giving people a chance to recover from honest mistakes, comparing it to how airline pilots double-check each other's altitude entries rather than treating every typo as unrecoverable. We also talked about when the opposite approach makes sense — some teams prefer to stop immediately on bad input because it should never have been possible to enter it in the first place.

## Rewriting with a Loop and Try/Except
Using Visual Studio Code, we rewrote the withdrawal program with a loop that keeps asking for input and a try/except block around it. We asked for an amount, tried to convert it to a whole number, and sent it to the withdraw method, which still raises an error for an amount that is not a multiple of 20 or an amount bigger than the balance. Instead of the program stopping, we caught that error, printed the message, and looped back to ask again. We tested this live with bad input like letters and near-miss numbers before landing on a valid withdrawal amount. Next week's cleanup assignment is to add similar retry handling for the other two problems, with a limit on how many tries someone gets.
