## Designing a GPA tracker from a vague request

We practiced turning a loose request — "track a student's running GPA, no course database needed" — into working code. With Cheska, Kandace, and Bella's help, we worked out the pieces: GPA is a number with one decimal place, it starts at 0 on day one, and it updates using a letter-to-number conversion (A = 4.0, B = 3.0, C = 2.0, D = 1.0, F = 0). We stressed validating assumptions like this grade scale with the client up front, even ones that seem obvious, rather than discovering a mistake partway through a project.

## Writing and testing enter_grade

We wrote a method called `enter_grade` that takes a letter grade and a student's current GPA, converts the letter to a number using a chain of if statements, and returns the new GPA by averaging the old GPA with the new grade. We ran through several test cases by hand — a 4.0 student getting another A stays at 4.0, a 4.0 student getting an F drops to 2.0, and so on.

## Catching bad input

With Bella and Erica's help, we found two problems: a lowercase letter (like `a` instead of `A`) fell through every check and silently scored as zero, and a mistyped letter (like `S`) did the same. We fixed the lowercase issue by converting input to uppercase, then discussed a more serious protection: raising a `ValueError` when the letter isn't A, B, C, D, or F, or when a starting GPA is negative — deliberately crashing the program rather than silently corrupting the GPA. We also saw Python's `in` operator as a compact way to check membership in a set of acceptable grades, and discussed keeping lines under about 80 characters for readability, a convention with roots in old 80-column punch cards.
