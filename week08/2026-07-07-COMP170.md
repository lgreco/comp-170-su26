## Workflow for the Quadratic Equation

We discussed our decision-making strategy when presented with a trio of values $(a,b,c)$ that correspond to the coefficients of the quadratic equation $ax^2+bx+c=0$.

The decision-making process can be visualized as a flow chart, below. In a flow chart, diamond shapes correspond to questions that can be answered with a yes or a no.

![](./quadratic_flow_chart.svg)

The decision process can also be documented as pseudocode.

$$
\begin{align*}
& \textbf{solve quadratic}\ \text{with coefficients}\ (a,b,c)  \\
& \qquad \textbf{if}\ a = 0 \\
& \qquad\qquad \textsf{This is no longer a quadratic equation.} \\
& \qquad\qquad \textsf{But it may be a linear equation.} \\
& \qquad\qquad \textbf{if}\ b \ne 0 \\
& \qquad\qquad\quad \textsf{Linear equation with solution}\ x = -c/b \\
& \qquad\qquad \textbf{else} \\ 
& \qquad\qquad\quad \textsf{Not an equation} \\
& \qquad \textbf{else} \\ 
& \qquad\qquad \textsf{This is a quadratic equation because}\ a\ne 0 \\ 
& \qquad\qquad \textsf{Compute discriminant}\ \delta = b^2-4ac \\ 
& \qquad\qquad \textbf{if}\ \delta < 0 \\
& \qquad\qquad\qquad x_{1,2} = -\dfrac{b}{2a}\pm \dfrac{\sqrt{-\delta}}{2a}{\color{red}i}\quad (\textsf{complex numbers}) \\ 
& \qquad\qquad \textbf{else} \\
& \qquad\qquad\qquad x_{1,2} = -\dfrac{b\pm\sqrt{\delta}}{2a}\quad (\textsf{real numbers})
\end{align*}
$$

Notice that the pseudocode has exactly one branch for every diamond in the flow chart above. This is not a coincidence: pseudocode and flow charts are two notations for the same decision tree, and `solve_quadratic()` in `mathemagics.py` is a third — the Python code that actually runs.

## Representing Complex Numbers

All that remains is how to represent complex numbers. As it happens, Python is one of the few programming languages with built-in support for such numbers. However, for illustrative purposes we will define our own representation.

A complex number is a mathematical entity of the form

$$
z = \alpha + {\color{red}i}\beta
$$

where $\color{red}i$ is the imaginary unit, and $\alpha$ and $\beta$ can be any real numbers. This unit is an artificial mathematical concept that corresponds to $\sqrt{-1}$, and it is defined as ${\color{red}i}^2=-1$. (The symbols $\alpha$ and $\beta$ above are not the same as $a$ and $b$ in the quadratic equation.)

In a complex number, $\alpha$ is called its **real part** and $\beta$ its **imaginary part.** In plain terms, a complex number is a *pair of a real part and an imaginary part.* An alternative notation for a complex number is

$$ z = (\alpha, \beta) $$

which lends itself nicely to a programmatic representation, either as a Python list:
```python
z = [alpha, beta]
```
or as a Python tuple:
```python
z = (alpha, beta)
```

`mathemagics.py` uses the tuple form: whenever `solve_quadratic()` returns a complex root, that root is a 2-element tuple `(real_part, imaginary_part)` rather than a single number. That is why the test files below check `x1[0]` and `x1[1]` separately when a root is expected to be complex.

## Testing `solve_quadratic()`: From Naive to Professional

Writing `solve_quadratic(a, b, c)` is only half the job. How do we convince ourselves — and anyone reading our code — that it is *correct*? We can't check every one of the infinitely many possible $(a,b,c)$ triples, but we can pick a handful of representative cases, one for each branch of the flow chart above, and check that the function returns what the math predicts.

This is **testing**: for a chosen input, we already know the expected output by hand (or by a formula), and we compare it against what the function actually returns. The three files below test the exact same eleven cases against the exact same function, `solve_quadratic()` in `mathemagics.py`. What changes across the three files is *how* the checking is organized — and that organization gets progressively less repetitive and more informative as we go.

| Case | Coefficients $(a,b,c)$ | What it exercises |
|---|---|---|
| 1–2 | $(0,0,0)$, $(0,0,5)$ | No equation / contradiction $\to$ `(NaN, NaN)` |
| 3–4 | $(0,2,4)$, $(0,3,-9)$ | $a=0$: linear equation, one repeated root |
| 5 | $(1,0,0)$ | Trivial quadratic, $x=0$ (double root) |
| 6 | $(1,0,-4)$ | $b=0$, positive discriminant $\to$ two real roots |
| 7 | $(1,0,4)$ | $b=0$, negative discriminant $\to$ complex roots |
| 8 | $(1,-3,2)$ | Two distinct real roots |
| 9 | $(1,2,1)$ | Discriminant $=0$ $\to$ repeated real root |
| 10 | $(1,2,5)$ | Complex roots with nonzero real part |
| 11 | $(-1,3,-2)$ | Negative $a$ (root order may swap) |

### Level 1 — Naive testing (`test_math_naive.py`)

This is the most direct thing you could write: for each case, call `solve_quadratic()`, write an `if`/`else` that compares the result to the expected value by hand, and `print` `PASS` or `FAIL`.

```python
x1, x2 = solve_quadratic(0, 2, 4)
if abs(x1 - (-2)) < 1e-9 and abs(x2 - (-2)) < 1e-9:
    print("PASS: 2x + 4 = 0 -> x1 = x2 = -2")
else:
    print("FAIL: 2x + 4 = 0 -> expected (-2, -2), got", (x1, x2))
```

This works — run `python3 test_math_naive.py` and you get eleven `PASS`/`FAIL` lines — but notice the cost:

- The expression `abs(x1 - (-2)) < 1e-9` (a **tolerance check**, needed because floating-point arithmetic is rarely *exactly* equal) is retyped by hand in every single test, with room for a typo each time.
- There is no function boundary around each case, so nothing stops one test's temporary variables from leaking into the next.
- The pass/fail bookkeeping (formatting the message, deciding what counts as close enough) is mixed in with the actual check, so the file gets longer and harder to scan the more cases you add.

Naive testing is a legitimate first step — it catches real bugs — but it does not scale past a handful of cases.

### Level 2 — Plain testing with functions (`test_math_plain.py`)

The next step is to notice the repetition in Level 1 and factor it out into ordinary functions:

```python
def close_enough(a: float, b: float, tolerance: float = 1e-9) -> bool:
    """Return True if two floats are close enough to be considered equal."""
    return abs(a - b) < tolerance

def check_close(description: str, actual: float, expected: float) -> None:
    """Compare actual and expected floats within a small tolerance."""
    if close_enough(actual, expected):
        print(f"PASS: {description}")
    else:
        print(f"FAIL: {description} -- expected {expected}, got {actual}")
```

Each of the eleven cases now lives inside its own function, e.g. `test_a_zero_linear_equation()`, and calls `check_close()` instead of retyping the tolerance logic:

```python
def test_a_zero_linear_equation():
    # 2x + 4 = 0 -> x = -2
    x1, x2 = solve_quadratic(0, 2, 4)
    check_close("2x + 4 = 0 -> x1 = -2", x1, -2)
    check_close("2x + 4 = 0 -> x2 = -2", x2, -2)
```

This is a real improvement over Level 1:

- The tolerance logic exists in exactly one place (`close_enough`), so fixing or tightening it fixes every test at once.
- Wrapping each case in its own function gives it a name and a private scope — no shared variables between cases.
- The file reads like a table of contents: the function names alone (`test_b_zero_complex_roots`, `test_negative_a`, ...) tell you what is covered.

But something new crept in: at the bottom of the file, under `if __name__ == "__main__":`, every test function has to be called *by hand*:

```python
test_a_zero_b_zero_c_zero()
test_a_zero_b_zero_c_nonzero()
test_a_zero_linear_equation()
...
```

Forget to add a line here for a new test function and it silently never runs — Python won't warn you. There is also still no summary at the end ("11 passed, 0 failed"); you have to scroll up and read every `PASS`/`FAIL` line yourself.

### Level 3 — `unittest` (`test-math.py`)

Python's standard library ships a testing framework, `unittest`, that solves exactly the two problems left over from Level 2: it *discovers and runs* every test automatically, and it *reports a summary*. At this level, the code is organized around a *class*, a program with object-oriented properties. This level is a bit advanced for us right now, but it is described here for future reference.

```python
import unittest
from mathemagics import solve_quadratic

class TestSolveQuadratic(unittest.TestCase):

    def test_a_zero_linear_equation(self):
        # 2x + 4 = 0 -> x = -2
        x1, x2 = solve_quadratic(0, 2, 4)
        self.assertAlmostEqual(x1, -2)
        self.assertAlmostEqual(x2, -2)

if __name__ == "__main__":
    unittest.main()
```

Two things replace what we wrote by hand in Levels 1 and 2:

- **`self.assertAlmostEqual(actual, expected)`** replaces `close_enough()` and the hand-rolled `if`/`else`. It is `unittest`'s built-in tolerance check for floating-point comparisons, so we no longer maintain our own.
- **`unittest.main()`** replaces the hand-written list of function calls at the bottom of the file. It scans the class for every method whose name starts with `test_` and runs each one automatically — add a new `test_...` method and it is picked up with zero extra bookkeeping.

Run it and you get output built for scanning, not reading line by line:

```
$ python3 test-math.py -v
test_a_zero_b_zero_c_zero ... ok
test_a_zero_b_zero_c_nonzero ... ok
...
----------------------------------------------------------------------
Ran 11 tests in 0.000s

OK
```

If a case fails, `unittest` isolates it — the other ten still run — and reports exactly which one failed, what was expected, and what was actually returned, without you writing a single `print`.

### Comparing the Three Levels

| | Naive (`test_math_naive.py`) | Plain (`test_math_plain.py`) | `unittest` (`test-math.py`) |
|---|---|---|---|
| Tolerance check | Retyped in every test | One shared `close_enough()` function | Built into `assertAlmostEqual` |
| Test isolation | None — one long script | One function per case | One method per case, run in isolation by the framework |
| Running the tests | Runs top to bottom automatically | Must call each `test_...()` by hand | `unittest.main()` discovers and calls every `test_...` method automatically |
| Result if you add a case but forget a step | N/A | Silently never runs if you forget to call it | Always runs — discovery is automatic |
| Summary at the end | None; read every line | None; read every line | `Ran 11 tests ... OK` (or a precise failure report) |

The math being tested never changes across the three files — only the amount of bookkeeping we ask Python to do for us. This is the same lesson as the flow chart at the top of this document: as a problem grows, we look for the representation (plain script $\to$ helper functions $\to$ a framework) that removes repeated, error-prone bookkeeping from the parts a human has to get right by hand.

### Try It Yourself

```bash

python3 test-math.py -v           # Level 3: unittest discovers and summarizes for you
```
